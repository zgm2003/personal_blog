在现代企业级应用中，数据列表分页是非常常见的功能。然而，随着数据量的急剧增加和业务场景的复杂化，传统的页码分页方式（offset + limit）在深分页（deep pagination）场景下，暴露出了严重的性能和一致性问题。尤其在操作日志、审计日志、消息流等"事件流"场景中，这个问题变得尤为突出。本文将从架构视角，深度剖析深分页问题及解决方案，并提供实战可落地的设计策略。

## 一、传统分页的局限性

传统的页码分页模型假设数据是静态数组：

```sql
SELECT * 
FROM operation_logs
ORDER BY created_at DESC
LIMIT 20 OFFSET 1000;
```

这种做法在浅分页（如前 5 页）时表现尚可，但一旦：

- 数据量非常大（百万级别以上）
- 用户需要查看深层历史记录（例如第 500 页）
- 数据表正在被实时写入

就会暴露出几个核心问题：

### 性能问题
MySQL 或 PostgreSQL 在 OFFSET 非常大的情况下，底层仍然需要扫描或跳过大量记录才能返回结果，导致查询时间呈线性增长，严重影响系统响应。

### 一致性问题
当表中有实时写入时，页码分页无法保证数据顺序的一致性。用户翻到第 N 页时，前 N-1 页的数据可能已被新插入的数据顶掉，导致用户看到重复或漏掉的记录。

### 事件连续性破坏
对于操作日志或审计日志场景，页码分页会让用户失去上下文。用户跳到第 50 页时，根本无法验证第 1–49 页的数据是否完整查看过，这在审计和合规场景下是不可接受的。

## 二、深分页下的游标分页（Cursor-based Pagination）

为解决上述问题，架构师通常会推荐游标分页（cursor pagination）。游标分页的核心思想是：

基于上一条记录的位置（通常用唯一排序字段和 ID 组合）进行下一页查询，而不是依赖页码和偏移量。

### 1. 核心实现方式

对于操作日志场景，可以使用 (created_at, id) 作为游标：

```sql
SELECT * 
FROM operation_logs
WHERE (created_at < :last_created_at)
   OR (created_at = :last_created_at AND id < :last_id)
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

每次加载更多时，将最后一条记录的 (created_at, id) 作为下次查询的游标，从而保证顺序的连续性和查询性能。

### 2. 游标分页的优点

- **性能稳定**：不管数据量有多大，查询都只依赖索引，不受 OFFSET 增长影响。
- **顺序一致**：即使有新日志插入，也不会破坏用户看到的连续事件流。
- **适合流式消费**：特别适合聊天记录、消息队列、SSE/实时流、操作日志等场景。

### 3. 游标分页的限制

- **不能直接跳页**：无法像页码那样从第 1 页直接跳到第 50 页，因为游标依赖上一页位置。
- **用户心智不匹配**：部分用户仍希望看到"第 N 页"，需要用时间或业务 ID 锚点替代。

## 三、操作日志的特殊场景分析

操作日志与一般 CRUD 列表的区别在于：

### 语义是事件流，而非状态表格

传统 CRUD 表格展示的是某个状态的快照，例如库存列表、订单列表。

日志记录的是事件序列，时间和顺序才是核心价值。

### 审计和合规要求

日志必须完整、可追溯，任何跳页都可能造成上下文丢失。

"看第 50 页"无法证明用户已经看过前 49 页数据，这是审计漏洞。

### 用户需求的误解

用户希望"跳页"看历史，其本质是"定位到某段时间 / 某个操作"，而非真正的页码。

## 四、深分页与日志查询的解决方案

结合业务需求和架构实践，操作日志深分页处理可以分为几个层次：

### 1. 时间锚点 + 游标分页（核心方案）

- **前端**：提供时间范围选择、操作人筛选、操作类型筛选。
- **分页逻辑**：游标分页 + "加载更多"按钮或无限滚动。
- **后端**：支持 (created_at, id) 游标查询，保证顺序与性能。

示例界面：

```
┌─────────────────────────────────────────┐
│  时间范围筛选：[2026-01-01] ~ [2026-01-21] │
│  操作人：[下拉选择]                        │
│  操作类型：[下拉选择]                      │
├─────────────────────────────────────────┤
│  ↑ 历史记录                              │
│  ─────────────────────────────────       │
│  事件 1                                  │
│  事件 2                                  │
│  事件 3                                  │
│  ...                                     │
│  ─────────────────────────────────       │
│  [加载更多] 或 无限滚动                   │
│  ↓ 最新记录                              │
└─────────────────────────────────────────┘
```

### 2. 前端分页交互设计原则

- 不显示总数（total）或者显示"近似数量"，防止误导用户。
- 去掉页码输入框，避免用户期望跳页功能。
- 提供"定位能力"，如按时间、操作人、业务 ID 搜索。

### 3. 后端 API 设计

统一分页接口 PageFetcher，支持两种模式：

- **状态列表（页码分页）**：适用于 CRUD / 管理后台浅分页。
- **事件流（游标分页）**：适用于日志、审计、消息流。

API 返回格式：

```typescript
interface PageFetcherResult<T> {
  list: T[];
  nextCursor?: string;
  hasMore: boolean;
}
```

游标可复用，实现上下翻、加载更多，而无需依赖页码。

## 五、深分页跳页的现实考量

虽然用户期望"跳到第 50 页"，但在日志场景中：

### 页码不存在稳定边界

数据实时变化，页码意义丢失

### 审计要求连续查看

真正的跳页需求可以通过 时间锚点 / 条件定位 替代，满足用户查历史的核心诉求，同时保持系统性能和数据一致性。

如果真的想做"页码跳页"的体验：

- 需要维护页码 → 游标映射表（状态化），成本高
- 易出错，可能与实时写入冲突
- 更适合静态数据报表，而非操作日志

## 六、架构层面总结

### 深分页本质
当数据量大、数据不断写入时，页码分页不可用。

### 游标分页适用场景
事件流、消息流、日志、聊天记录、操作流水。

### 日志与表格的区别

- **表格**：状态快照、页码分页适用
- **日志**：事件流、游标分页适用

### 前端 UX 指南

- 不显示页码，总数可选
- 提供时间 / 条件筛选
- 支持加载更多或无限滚动

### 后端 API 指南

- 统一分页接口，区分状态列表和事件流
- 游标以 (created_at, id) 为核心

## 七、技术实现示例

以下是一个完整的游标分页实现示例（PHP + Laravel Eloquent）：

```php
// BaseDep.php 中的游标分页实现
public function listCursor(array $param, ?callable $queryBuilder = null, array $columns = ['*'], bool $checkDel = true): array
{
    $pageSize = $param['page_size'] ?? 20;
    $cursor = $param['cursor'] ?? null;
    
    $query = $this->model->select($columns);
    
    if ($checkDel) {
        $query->where('is_del', CommonEnum::NO);
    }
    
    if ($cursor) {
        $query->where('id', '<', $cursor);
    }
    
    if ($queryBuilder) {
        $queryBuilder($query);
    }
    
    $list = $query->orderBy('id', 'desc')->limit($pageSize + 1)->get();
    
    $hasMore = $list->count() > $pageSize;
    if ($hasMore) {
        $list = $list->take($pageSize);
    }
    
    return [
        'list' => $list,
        'next_cursor' => $hasMore ? $list->last()->id : null,
        'has_more' => $hasMore
    ];
}
```

## 八、结语

深分页问题不仅是 SQL 查询效率问题，更是业务模型与分页模型匹配的架构问题。在操作日志、审计日志等事件流场景中：

- 强行用页码分页会破坏连续性、审计能力和数据一致性
- 游标分页结合时间锚点和条件筛选，才是长期可维护、性能稳定、用户体验合理的方案
- 通过分离 状态表格 与 事件流组件，前后端分页逻辑复用，同时保证语义清晰、用户可用、架构稳健，才能构建企业级日志查询系统。

### ✅ 最终建议：

- 操作日志页面单独做流式组件（EventStream / LogTimeline）
- 底层复用分页逻辑（PageFetcher）
- UI 上提供时间锚点定位、操作人/类型筛选、加载更多
- 不使用页码跳转，保证审计连续性和系统性能

游标分页不仅仅是技术优化，更是对业务场景的深刻理解。当我们认识到日志是"事件流"而非"状态表格"时，就能做出更符合业务本质的技术决策。
